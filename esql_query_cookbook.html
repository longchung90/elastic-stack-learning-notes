1 — Response codes what is your error rate?
Error rate as a percentageEVAL + STATS
Calculate what percentage of all requests are errors. EVAL creates a new computed column on each row before STATS aggregates it.
FROM kibana_sample_data_logs
| EVAL is_error = CASE(response.keyword >= "500", 1, 0)
| STATS
    total    = COUNT(*),
    errors   = SUM(is_error),
    error_pct = ROUND(100.0 * SUM(is_error) / COUNT(*), 2)
Why response.keyword? Because response is mapped as text — we have to use the keyword sub-field for comparisons. In a proper integer mapping you would write response >= 500 directly.
Response code breakdownSTATS + SORT
See every distinct response code and how often it appeared. Good first query to run on any web log dataset.
FROM kibana_sample_data_logs
| STATS count = COUNT(*) BY response.keyword
| SORT count DESC
This is where the response as text problem becomes visible — codes come back as "200", "404" etc. Notice you cannot do >= here.
2 — Traffic patterns when and from where?
Requests per hour of dayDATE_TRUNC
DATE_TRUNC rounds @timestamp down to the nearest bucket. This shows you peak traffic hours — essential for capacity planning.
FROM kibana_sample_data_logs
| STATS requests = COUNT(*)
    BY hour = DATE_TRUNC(1 hour, @timestamp)
| SORT hour ASC
Change "1 hour" to "1 day" or "15 minutes" to zoom in or out on the time grain.
Top source countriesSTATS BY geo
Aggregate on the geo.src keyword field. Shows where your users are coming from — useful for CDN routing decisions.
FROM kibana_sample_data_logs
| STATS requests = COUNT(*) BY geo.src
| SORT requests DESC
| LIMIT 10
Top traffic corridors (src → dest)multi-field group
Group by the srcdest pair field to see which country-to-country routes carry the most traffic.
FROM kibana_sample_data_logs
| STATS requests = COUNT(*),
        total_bytes = SUM(bytes)
    BY geo.srcdest
| SORT requests DESC
| LIMIT 10
srcdest is pre-combined as "US:IN" — this avoids needing a string concat across two fields.
3 — Bytes and bandwidth how heavy are requests?
Bandwidth stats per file extensionpercentiles
See which file types are heaviest. ZIP downloads vs CSS files will look very different. PERCENTILE shows the 95th percentile — the "slow tail" of your distribution.
FROM kibana_sample_data_logs
| STATS
    requests  = COUNT(*),
    avg_bytes = ROUND(AVG(bytes), 0),
    p95_bytes = PERCENTILE(bytes, 95),
    max_bytes = MAX(bytes)
  BY extension.keyword
| SORT avg_bytes DESC
Heavy requests — find the outliersWHERE + SORT
Find individual large responses. Useful for catching accidental large payload returns or debugging slow requests.
FROM kibana_sample_data_logs
| WHERE bytes > 50000
| KEEP @timestamp, request, bytes, response.keyword, clientip
| SORT bytes DESC
| LIMIT 20
KEEP selects only the columns you want — makes the output readable instead of returning all 20+ fields.
4 — EVAL — creating new fields on the fly computed columns
Classify requests by size bucketEVAL + CASE
CASE is ES|QL's if-then-else. EVAL adds a new computed column to every row. Here we classify each request into a size band without changing the underlying data.
FROM kibana_sample_data_logs
| EVAL size_band = CASE(
    bytes < 1000,   "small  < 1kb",
    bytes < 10000,  "medium 1–10kb",
    bytes < 100000, "large  10–100kb",
                    "huge   > 100kb"
  )
| STATS count = COUNT(*) BY size_band
| SORT count DESC
Extract day of week from timestampDATE_FORMAT
DATE_FORMAT extracts parts of a timestamp as strings. This reveals weekday vs weekend traffic patterns — critical for on-call scheduling.
FROM kibana_sample_data_logs
| EVAL day = DATE_FORMAT("EEEE", @timestamp)
| STATS requests = COUNT(*) BY day
| SORT requests DESC
EEEE = full day name (Monday, Tuesday...). Use MM for month, HH for hour, yyyy for year.
KB instead of bytes — human readableEVAL math
EVAL can do arithmetic. Divide bytes by 1024 to get KB. ROUND cleans up the decimal places.
FROM kibana_sample_data_logs
| EVAL kb = ROUND(bytes / 1024.0, 2)
| STATS
    avg_kb = ROUND(AVG(kb), 2),
    max_kb = ROUND(MAX(kb), 2)
  BY extension.keyword
| SORT avg_kb DESC
| LIMIT 10
5 — WHERE filters — narrowing before aggregating pre-filter patterns
Only errors, broken down by hostWHERE + STATS
Filter first with WHERE, then aggregate. This is cheaper than aggregating everything and filtering the result — ES applies WHERE before loading documents into the pipeline.
FROM kibana_sample_data_logs
| WHERE response.keyword == "503"
    OR response.keyword == "404"
    OR response.keyword == "500"
| STATS errors = COUNT(*) BY host.keyword, response.keyword
| SORT errors DESC
| LIMIT 20
Specific time windowWHERE time range
Filter to a specific time slice inside ES|QL — combine with the Kibana time picker or use standalone in Dev Tools.
FROM kibana_sample_data_logs
| WHERE @timestamp >= "2026-04-14T00:00:00Z"
    AND @timestamp <  "2026-04-15T00:00:00Z"
| STATS
    requests = COUNT(*),
    errors   = COUNT(*) WHERE response.keyword != "200"
  BY host.keyword
| SORT requests DESC
COUNT(*) WHERE is a conditional count — counts only rows matching the inner condition. Added in recent ES|QL versions.
LIKE — pattern matching on stringsLIKE wildcard
LIKE does wildcard matching — % means "any characters". Useful when you know part of a URL path but not the full value.
FROM kibana_sample_data_logs
| WHERE request LIKE "%elasticsearch%"
| STATS
    downloads = COUNT(*),
    total_mb  = ROUND(SUM(bytes) / 1048576.0, 2)
  BY request.keyword
| SORT downloads DESC
| LIMIT 10
6 — Chaining pipes — building complex pipelines multi-step
Top IPs with high error ratespipe chain
Each | pipe passes its output as input to the next stage. This builds a multi-step analysis in one query — no subqueries needed.
FROM kibana_sample_data_logs
| STATS
    total  = COUNT(*),
    errors = COUNT(*) WHERE response.keyword != "200"
  BY clientip
| EVAL error_rate = ROUND(100.0 * errors / total, 1)
| WHERE total > 10
| SORT error_rate DESC
| LIMIT 15
WHERE total > 10 removes IPs with only 1–2 requests — they would show 100% error rates by chance and are noise.
Bandwidth heatmap by hour + extensiontwo-dimension group
Group by two dimensions simultaneously — time bucket AND file type. Shows which file types drive bandwidth at which hours.
FROM kibana_sample_data_logs
| STATS total_mb = ROUND(SUM(bytes) / 1048576.0, 2)
    BY
      hour = DATE_TRUNC(1 hour, @timestamp),
      extension.keyword
| WHERE total_mb > 1
| SORT hour ASC, total_mb DESC
RENAME + DROP — tidy output columnsoutput shaping
RENAME renames columns in the output. DROP removes columns you do not need. Makes results readable in Dev Tools and dashboards.
FROM kibana_sample_data_logs
| STATS
    req   = COUNT(*),
    bytes = SUM(bytes)
  BY geo.src
| RENAME geo.src AS country,
         req     AS requests,
         bytes   AS total_bytes
| EVAL total_mb = ROUND(total_bytes / 1048576.0, 2)
| DROP total_bytes
| SORT requests DESC
| LIMIT 10
